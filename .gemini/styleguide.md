# 코드 리뷰 가이드라인

이 프로젝트는 Unity를 사용한 게임 개발 프로젝트입니다. Gemini는 아래 지침에 따라 코드 리뷰를 수행해 주세요.

## 1. 언어 설정

- 모든 리뷰 댓글, 요약, 제안은 반드시 **한국어**로 작성해 주세요.
- 전문 용어는 관용적인 표현(예: 캐싱, 인스턴스화 등)을 사용하되, 설명은 한국어로 합니다.

## 2. 리뷰 중점 사항

- **성능 최적화**: 유니티의 `Update()` 내에서의 가비지 컬렉션 발생이나 비효율적인 연산을 체크해 주세요.
- **가독성 및 규칙**: C# 코딩 컨벤션(PascalCase 등)이 잘 지켜졌는지 확인해 주세요.
- **로직 오류**: 발생 가능한 널 참조(Null Reference)나 논리적 오류를 지적해 주세요.

## 3. 답변 톤앤매너

- 시니어 개발자가 주니어 개발자에게 조언하듯 **친절하고 건설적인 톤**으로 작성해 주세요.
- 단순히 "틀렸다"고 하기보다 "이렇게 개선하면 성능/가독성 면에서 더 좋을 것 같습니다"라고 제안해 주세요.

## 4. 답변 형식 (필수 준수 사항)

- **구체적인 위치 명시**: 수정을 제안할 때는 반드시 해당 **파일명**과 **줄 번호(Line Number)** 또는 근처의 문맥을 정확히 언급해 주세요.
- **코드 대치 가이드**: 코드를 어떻게 바꿔야 할지 알려줄 때는 아래와 같은 형식을 사용해 주세요:
  - **기존 코드 (As-is)**: [수정 전 코드 조각]
  - **수정 코드 (To-be)**: [수정 후 코드 조각]
- **단계별 설명**: 복잡한 수정이 필요한 경우, 개발자가 따라 하기 쉽도록 Step 1, Step 2 순서로 나누어 상세히 설명해 주세요.
- **유니티 안정성 검토**:
  - 코드를 수정할 때 유니티 에디터에서 **컴파일 오류**가 발생하지 않도록 문법적 정확성을 검토해 주세요.
  - 가급적 유니티의 생명주기(Life Cycle)와 물리 엔진 규칙을 준수하는 코드를 제안해 주세요.
- **인스펙터(Inspector) 가이드**:
  - 코드 수정 후 인스펙터 창에서 추가로 설정해야 하는 부분(예: Reference 할당, SerializeField 값 변경, 컴포넌트 추가 등)이 있다면 반드시 함께 명시해 주세요.
- **커밋(Commit) 가이드**:
  - 코드 수정 후 Commit을 할 경우 커밋 메시지도 추천해 주세요. 커밋 메시지는 영어를 선호합니다.
